# GuardsTracker – AI Agent Context Guide

## Project Overview
**GuardsTracker** is a browser-based multiplayer initiative tracker for **Guards of Atlantis II** (a tabletop game). It enables:
- **Host** to create a room, manage game state, and broadcast updates
- **Players** to join a session, pick a character, and participate in turns
- **Offline mode** to test locally without a server

**Technology Stack**: Vanilla JavaScript (ES5+), Socket.IO, localStorage, no build tools required.
**Deployment**: Client hosted on **GitHub Pages** (`https://hoeleboele.github.io`); Socket.IO server hosted on **Render** (`https://goainittracker.onrender.com`).

## Architecture

### Module Organization
The codebase follows **SOLID principles** with a namespace pattern (`window.GoA`). 11 focused modules handle distinct concerns:

| Module | Responsibility | Key Exports |
|--------|-----------------|-------------|
| **constants.js** | Named constants (SERVER_URL, RECONNECT_MAX_AGE, VIEWS array, etc.) | `GoA.VIEWS`, `GoA.SERVER_URL`, ability names |
| **state.js** | Mutable game state (gameMode, players, turns, currentPlayerId, etc.) | `GoA.state.*` properties |
| **utils.js** | DOM helpers, general utilities ($, $$, genCode, toast notifications) | `GoA.$()`, `GoA.genCode()`, `GoA.toast()` |
| **storage.js** | localStorage reconnect persistence (4-hour expiry) | `GoA.save()`, `GoA.load()` |
| **characters.js** | 42-character roster, lookups, abilities metadata | `GoA.CHARACTERS`, `GoA.getCharacter()`, ability reference |
| **render.js** | **ALL DOM rendering** (phase dispatcher, player/turn lists, offline setup UI) | `GoA.render()` dispatches to phase-specific functions |
| **ui.js** | View switching, character picker, character theming, UI state | `GoA.showView()`, `GoA.pickCharacter()`, theme functions |
| **turns.js** | Complete turn lifecycle (initiative, advancement, mixed-ties, rounds) | `GoA.calculateInitiative()`, `GoA.nextTurn()`, turn logic |
| **abilities.js** | All 5 character abilities (Hanu, Ignatia, Tigerclaw, Takahide, Tali) | Ability handlers with signature: `(targetId[, penalty])` |
| **network.js** | Socket.IO connections, broadcast, host/player message handlers | `GoA.connect()`, `GoA.broadcast()`, socket event handlers |
| **events.js** | **ALL event listener wiring** and application bootstrap | Runs on DOMContentLoaded; no exports—just initialization |

### Load Order (in index.html)
Scripts load in dependency order. **Critical**: constants → state → utils → storage → characters → render → ui → turns → abilities → network → events.

### Key Patterns

**Lazy Cross-Module Calls**: Calls between modules happen inside function bodies (e.g., `GoA.broadcast()` called from turn logic), never at module load time. This allows safe circular dependencies.

**Uniform Ability Signature**: All ability handlers accept `(targetId[, penalty])` for consistency.

**No DOM Logic Outside render.js & ui.js**: Game logic (turns, abilities, state) never directly manipulates DOM.

**No Game Logic in render.js & ui.js**: UI modules call rendering/switching functions but don't make game decisions.

## State Management

**GameMode**: "host", "player", or "offline" (set during initialization).

**Key State Properties** (in `GoA.state`):
- `gameMode` – Current mode
- `players` – Object map `{ [id]: { id, name, team, character, initiative, isConnected, disconnectedAt, ... } }`
- `turns` – Array of turn slot objects built by `revealTurns()` on the server
- `currentTurnIndex` – Index into the turns array for the active slot
- `initiativeToken` – `'blue'` or `'orange'` — which team goes first on tied initiative
- `mixedTies` – Object tracking tie-breaking pools per initiative value
- `reverseInitiative` – Boolean; when true, lower initiative goes first
- `usedAbilities` – Array of ability keys consumed this round
- `phase` – Server-side phase: `'lobby'`, `'initiative'`, `'turns'`, `'round-complete'`
- `serverConnected` – Boolean for socket connection status

**Key Constants** (in `GoA`):
- `GoA.SERVER_URL` – `'https://goainittracker.onrender.com'`
- `GoA.RECONNECT_KEY` / `GoA.RECONNECT_MAX_AGE` – localStorage key + 4-hour expiry
- `GoA.DISCONNECT_GRACE_MS` – 5 minutes; how long a disconnected player still blocks the round
- `GoA.LAST_NAME_KEY` – localStorage key for persisting the player's last-used name
- `GoA.VIEWS` – `['viewOfflineSetup', 'viewLobbyHost', 'viewInitiative', 'viewTurns', 'viewRoundComplete']`

**State Updates**: Direct assignment followed by render calls and `broadcastState()` from `server.js`.

## Network Model

**Host** → Creates a room on the server, manages game state, triggers turn advancement.
**Players** → Join a room via session code, listen for `game_state` broadcasts, submit initiative and abilities.
**Offline Mode** → No Socket.IO; local-only state updates and rendering.
**Server** → Stateful Node.js process (Express + Socket.IO) hosted on Render; owns `rooms[code]` and authoritative `gameState`; CORS-restricted to `https://hoeleboele.github.io`.

**Key Socket Events**:
- `game_state` – Server broadcasts full `gameState` to all clients in a room
- `player_joined` / `player_left` – Player connection lifecycle
- `submit_initiative` – Client sends initiative value
- `advance_turn` – Host advances to next turn slot
- `use_ability` – Client triggers a character ability
- `reconnect` – Client reconnects using stored session code

## Game Flow

1. **Landing**: User chooses Host, Player, or Offline.
2. **Setup Phase**:
   - Host: Add players, pick characters
   - Players: Wait for host to start
3. **Turn Phase**:
   - Host calls `GoA.nextTurn()` to advance turn
   - Current player's abilities trigger via event listeners
   - Abilities call ability handler from `abilities.js`
   - State updates, broadcasts, re-renders
4. **Round End**: Triggers after all players complete a round
5. **Game End**: Triggered when player count reaches 1 (winner) or game is manually ended

## Adding New Features

### Adding a New Character Ability
1. **Define metadata** in `characters.js` (ability name, description, requirements)
2. **Implement handler** in `abilities.js` with signature: `(targetId[, penalty])`
3. **Add event listener** in `events.js` to bind ability button to handler
4. **Test** via offline mode or browser console

### Adding a New View or UI State
1. **Add constant** to `constants.js` (VIEWS array)
2. **Add render function** to `render.js` (e.g., `renderCharacterPicker()`)
3. **Add display logic** to `ui.js` (e.g., `showView('characterPicker')`)
4. **Add CSS classes** to `styles.css`

### Adding Game Logic (Initiative, Turns, Phases)
1. **Add state** to `state.js`
2. **Implement logic** in `turns.js`
3. **Add broadcast calls** to `network.js` message handlers
4. **Add render dispatcher** in `render.js` to update UI

## Testing & Debugging

**Browser Testing**:
1. Open `index.html` in a modern browser
2. Open DevTools (F12) to check console for errors
3. Test each mode: Host → Player (via Socket.IO or same window); Offline

**Console Logging**:
- All state changes log to console in debug mode
- Add `console.log(GoA.state)` to inspect current state
- Use `GoA.toast()` for user-facing alerts

**Offline Mode**:
- Fastest way to test without server setup
- Host and player logic run locally in same browser
- No network latency; instant feedback

## Key Conventions

- **Namespace**: All public code lives under `window.GoA`
- **Private functions**: Prefix with `_` inside modules (e.g., `_renderPlayerList()`)
- **Event listeners**: Added in `events.js` only; never inline in HTML
- **Phase logic**: All phase transitions via `GoA.state.currentPhase` updates + render calls
- **Error handling**: Use `GoA.toast()` for user feedback; console.error for debugging
- **Server**: Optional; offline mode works without it for testing

## Common Tasks for AI Agents

| Task | File(s) | Approach |
|------|---------|----------|
| Fix rendering bug | `render.js`, `ui.js` | Find render function; check DOM selectors; verify state input |
| Add ability | `abilities.js`, `events.js`, `characters.js` | Implement handler → wire listener → define metadata |
| Debug turn logic | `turns.js`, `state.js` | Trace `nextTurn()` call; inspect state.turns array; check phase transitions |
| Network issue | `network.js`, `Server/server.js` | Check Socket.IO events; verify message format; test offline mode first |
| UI/styling | `styles.css`, `patch-colors.js`, `ui.js` | Use browser DevTools; trace CSS classes; check character theme logic |
| Performance | `render.js`, `events.js` | Profile with DevTools; batch DOM updates; avoid repeated renders |

## Known Limitations & Future Work

- **No persistent server data** (game state lost if Render instance restarts)
- **Multi-room support**: Server supports multiple rooms via `rooms[code]`, but UI only manages one active session
- **No undo system** (turns advance immediately)
- **Character art**: Avatars stored locally in `avatars_full/`; no CDN integration
- **Disconnect grace**: Disconnected players block round completion for up to 5 minutes (`DISCONNECT_GRACE_MS`)
